Auth Flow Spec
Google OAuth: Redirect flow — mở Google consent screen, callback về /auth/google/callback, tạo session, redirect /app hoặc /onboarding.
Email + Password: Standard form — email input + password input + "Quên mật khẩu?" flow (gửi reset link qua email). Register flow: email + password + confirm → verify email → session. Không dùng phone OTP, không dùng magic link.
Password reset: User nhập email → hệ thống gửi link reset (expire 1h) → user set password mới → redirect /app.
New user: Sau auth thành công → redirect /onboarding (S-01 splash). Existing user → redirect /app (swipe deck).
✦ v4.3 Fix H3 — Post-Auth Routing Decision (required spec for dev):
Decision gate after successful auth:
→ if user.onboarding_complete == true → redirect /app/home (S-05 Home, skip all onboarding)
→ if user.onboarding_complete == false && user.onboarding_step == null → redirect /onboarding (S-01 Splash, fresh start)
→ if user.onboarding_complete == false && user.onboarding_step != null → redirect /onboarding/step/{user.onboarding_step} (resume from last saved step — data already persisted locally per step)
Cross-device: onboarding_step is stored server-side — resuming from a new device works correctly.
DB flag: user.onboarding_complete is set to true only after user passes S-04b CTA ("Bắt đầu 14 ngày dùng thử →"). Reaching S-04 and skipping does NOT set this flag.
Session: JWT token, expire 30 ngày, refresh tự động. Store trong httpOnly cookie — không localStorage.

✦ v4.3 Fix H4 — "?" Defer Button Spec (4 edge cases):
What defer does: Moves current card to end of deck for current slot. Shows next card immediately.
Edge case 1 — Re-appearance: Deferred card re-appears exactly once per session, after all other active cards in the same slot have been seen. It does NOT re-defer a second time — on second encounter, only ✓ and ✕ are available.
Edge case 2 — All-deferred deck: If every card in a slot has been deferred and none confirmed → treat as empty deck → show S-05e empty state with 3 escape hatches. Same behavior as 0-result filter.
Edge case 3 — Memory Engine signal: Defer = neutral signal. Does NOT write to preference model (not positive, not negative). Only ✓ (accept) and ✕ (explicit reject) write to history. Rationale: defer means "not now," not "don't like."
Edge case 4 — Session end with deferred cards: Deferred cards are discarded at session end (not persisted). They reappear in next session's fresh deck as normal candidates. No "deferred" state carries over.

Data Model — Meal Structure
meal_structure[] — array of { meal_type, slots: [{ slot_order: int, dish_type: "savory"|"soup"|"vegetable"|"starch"|"salad"|"dessert", is_auto: bool }] }
Cơm/tinh bột có flag is_auto = true → không cần swipe, app suggest mặc định (cơm trắng, gạo lứt, bún...) hoặc user chỉnh nhanh sau summary.
Số lần swipe = số slot có is_auto = false. 4 người, 2 mặn + 1 canh + 1 rau = 4 swipes. Cơm auto → không tính.
User có thể override structure ngay trong swipe session bằng nút "Thêm 1 món" hoặc "Bỏ slot này" — không cần vào settings.
Preset system: user lưu tối đa 5 preset cấu trúc. Có thể switch preset từ home screen trước khi bắt đầu swipe.

Design Decisions
Toggle vị trí: Nằm giữa Memory Engine signal và swipe card — không phải top nav, không phải settings. Discoverable nhưng không cạnh tranh với primary flow.
Toggle ON state: Background xanh lá nhạt + border xanh + icon đổi màu + hiện tên 5 nguyên liệu đã chọn dưới label. User biết ngay filter đang active và với những gì.
NLP input — Option A (text dominant): Text field xuất hiện đầu tiên trong bottom sheet, trên chips. Placeholder: "nửa con gà, cà chua, mấy quả trứng..." → debounce 600ms → POST /api/pantry/parse → Gemini 2.5 Flash → chip auto-select. Confidence ≥0.7: chip dark (xác nhận). Confidence 0.4–0.7: chip amber border ("? Gừng") — caret icon visible, user tap để confirm hoặc bỏ qua. Confidence <0.4: không pre-select. Text field và chips merge realtime: type thêm → parse lại → add vào selection đã có. Fallback nếu LLM fail: text field ẩn spinner, chips vẫn dùng được bình thường (graceful degrade, không báo lỗi cho user).
Chip selection logic: 3 nhóm: Thịt/Cá, Rau/Củ/Đậu, Trứng/Khác. Default hiện ~18 chips phổ biến nhất. "Thêm khác..." → inline search không navigate đi đâu. Không cần scroll nếu nguyên liệu thuộc top 18. Chips là secondary path — dùng khi user muốn browse hoặc correct kết quả NLP.
Delta indicator trên card: Tầng 0 (nấu được ngay) → green pill "✓ Nấu được ngay". Tầng 1 (mua thêm 1 thứ) → amber pill "Mua thêm: [tên nguyên liệu]". Tầng 2+ → không hiện trong deck (ẩn để giữ deck quality). Sort: Tầng 0 → Tầng 1. Health filter vẫn cứng — không có món nào không an toàn xuất hiện bất kể delta.
S-07 Ingredient list: Full list nguyên liệu của tất cả món trong bữa — không lọc. Đã có trong tủ → text-decoration:line-through + màu #bbb + check icon. Cần mua → nền amber nhạt + bold + dot đậm. Header badge "Cần mua: X thứ" — số này là leading metric của feature.
✦ v4.1 · Persistence: PERSIST (thay đổi từ session-only). Tủ lạnh được lưu vào DB per user. Rationale: session-only mâu thuẫn với Family Memory Engine moat — nếu app "nhớ" sở thích nhưng không nhớ nguyên liệu, inconsistency sẽ phá trust. Implementation: (1) pantry_items[] table per user_id, (2) "Tủ lạnh của tôi" tab trong S-11 Settings với bulk clear button + last-updated timestamp, (3) Gentle nudge sau 3 ngày không update: "Tủ lạnh có thể đã thay đổi — cập nhật để gợi ý chính xác hơn". Data quality risk: user không update → pantry stale → recommendation sai → trust bị phá. Mitigate bằng timestamp visible và nudge logic, không bằng automation. Toggle OFF khi pantry > 5 ngày chưa update — hiện empty state khuyến khích reset.
Data cần build: Mỗi món cần có: (a) danh sách nguyên liệu bắt buộc vs tuỳ chọn, (b) coverage score algorithm: user có ≥80% nguyên liệu bắt buộc = Tầng 0. Một nguyên liệu thiếu = Tầng 1. Không phải tech phức tạp — là data modelling work cho 500 món.

Design Principles — Rewards System
Recognition về gia đình, không về app: "18 ngày chăm sóc gia đình" — không phải "Bạn đã dùng app 18 ngày". Copy, data, visual đều nói về kết quả của gia đình, không về engagement với sản phẩm.
Weekly Card trigger: Chủ Nhật 20:00 — background job. Chỉ generate nếu user có ≥ 3 bữa trong tuần. Dưới 3 bữa → không generate (card trống trông tệ + pressure khi user bận). Delivered via push notification (nếu opted-in) hoặc banner trên S-05 Home.
Monthly Report trigger: Ngày cuối tháng 20:00. Minimum ≥ 10 bữa trong tháng. Render server-side PNG → lưu R2 → URL trong family_reports table. Image immutable sau khi generate.
Caption UX: Default auto-generated từ template (không LLM): "[X] bữa tháng [N] — [family name nếu có] đang được chăm sóc". User tap caption field → editable inline. Share button active bất kể caption có được sửa hay không.
Share targets: Facebook (Open Graph image share) + Zalo (Web Share API). Fallback: copy-to-clipboard + download PNG. Không yêu cầu login hay permission mới — dùng Web Share API nếu available.
Sổ Gia Đình access: S-11 Settings → "Sổ Gia Đình". Không có bottom nav tab riêng (tránh chiếm không gian). Empty state: hidden hoàn toàn cho đến khi tháng đầu tiên complete.
❌ Không build (cùng Phase): Streak counter nổi bật · Push "Chưa plan bữa tối" · Badge/điểm · Rating ngay sau chọn bữa · Animation >300ms trên swipe. Lý do: tạo anxiety, guilt-trip, noise — không phù hợp identity người chăm sóc gia đình.
